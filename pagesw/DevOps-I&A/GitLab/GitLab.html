📚 Introduction

What is GitLab?

GitLab is a complete DevOps platform delivered as a single application. It provides Git repository management, issue tracking, code review, CI/CD pipelines, container registry, and more - all in one unified interface. Unlike other tools that require multiple integrations, GitLab provides the entire DevSecOps lifecycle out of the box.

Key Features

📦 Source Code Management — Git repositories with powerful merge requests
🔄 CI/CD Pipelines — Built-in continuous integration and deployment
🐛 Issue Tracking — Advanced issue boards and planning tools
📦 Container Registry — Built-in Docker container registry
🔒 Security Scanning — SAST, DAST, dependency scanning
☸️ Kubernetes Integration — Native k8s deployment support
📊 Analytics — DevOps metrics and insights
🤝 Collaboration — Code review, wikis, snippets

Why Choose GitLab?

  • All-in-One Platform: Complete DevOps lifecycle in a single application.
  • Built-in CI/CD: No need for separate CI/CD tools like Jenkins or CircleCI.
  • Self-Hosted Option: Full control with GitLab CE (Community Edition) or EE (Enterprise Edition).
  • Auto DevOps: Automatic CI/CD configuration for standard applications.
  • Powerful Merge Requests: Advanced code review with inline comments and approvals.
  • Kubernetes Native: First-class support for Kubernetes deployments.

Use Cases

💻 Software Development — Full DevOps lifecycle
🔄 CI/CD Automation — Automated testing and deployment
🔒 DevSecOps — Integrated security scanning
☁️ Cloud Native — Kubernetes and container workflows

GitLab Account Setup

Choose between GitLab.com (cloud-hosted) or self-hosted GitLab.

# Create free account at gitlab.com

# Access via: https://gitlab.com

Install GitLab Runner (for CI/CD)

# Linux (Debian/Ubuntu)
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | sudo bash
sudo apt-get install gitlab-runner

# macOS
brew install gitlab-runner

# Windows (PowerShell as Administrator)
New-Item -ItemType Directory -Force -Path "C:\GitLab-Runner"
cd C:\GitLab-Runner
Invoke-WebRequest -Uri "https://gitlab-runner-downloads.s3.amazonaws.com/latest/binaries/gitlab-runner-windows-amd64.exe" -OutFile "gitlab-runner.exe"
.\gitlab-runner.exe install
.\gitlab-runner.exe start

# Register runner
gitlab-runner register
# Follow prompts: enter GitLab URL, token (from Settings > CI/CD > Runners)

Install GitLab CLI (glab)

# macOS
brew install glab

# Windows
winget install -e --id GLab.GLab

# Linux
# Download from: https://gitlab.com/gitlab-org/cli/-/releases

# Authenticate
glab auth login

SSH Key Setup

# Generate SSH key
ssh-keygen -t ed25519 -C "your_email@example.com"

# Start SSH agent
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

# Copy public key
cat ~/.ssh/id_ed25519.pub

# Add to GitLab: Profile > Preferences > SSH Keys

Git Configuration

# Configure Git
git config --global user.name "Your Name"
git config --global user.email "your_email@example.com"

# Set GitLab as default remote
git remote add origin git@gitlab.com:username/repo.git

📖 User Guide

Creating and Managing Repositories

# Create new project via CLI
glab repo create my-project --public

# Clone repository
git clone git@gitlab.com:username/my-project.git
cd my-project

# Create and push first commit
echo "# My Project" > README.md
git add README.md
git commit -m "Initial commit"
git push -u origin main

Branches and Merge Requests

# Create feature branch
git checkout -b feature/new-feature

# Make changes and commit
git add .
git commit -m "feat: add new feature"
git push -u origin feature/new-feature

# Create merge request via CLI
glab mr create --fill --yes

# Or create via web: Project > Merge Requests > New merge request

# Review and merge
glab mr view
glab mr merge

Issues and Boards

# Create issue
glab issue create --title "Bug: fix login error" --description "Users cannot login" --label bug

# List issues
glab issue list

# Close issue
glab issue close 123

# Create issue board: Project > Issues > Boards

Container Registry

# Login to GitLab container registry
docker login registry.gitlab.com -u your-username

# Build and tag image
docker build -t registry.gitlab.com/username/project/app:latest .

# Push to registry
docker push registry.gitlab.com/username/project/app:latest

# Pull image
docker pull registry.gitlab.com/username/project/app:latest

🔄 CI/CD Pipelines

Basic .gitlab-ci.yml

GitLab CI/CD uses .gitlab-ci.yml file in the repository root to define pipelines.

# .gitlab-ci.yml - Basic Pipeline
stages:
  - build
  - test
  - deploy

build-job:
  stage: build
  script:
    - echo "Building the application..."
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test-job:
  stage: test
  script:
    - echo "Running tests..."
    - npm test
  coverage: '/Coverage: \d+\.\d+%/'

deploy-job:
  stage: deploy
  script:
    - echo "Deploying application..."
    - ./deploy.sh
  only:
    - main

Node.js CI/CD Pipeline

image: node:20

stages:
  - install
  - test
  - build
  - deploy

cache:
  paths:
    - node_modules/

install:
  stage: install
  script:
    - npm ci

test:
  stage: test
  script:
    - npm run test
    - npm run lint

build:
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

deploy:production:
  stage: deploy
  script:
    - npm run deploy
  environment:
    name: production
    url: https://myapp.com
  only:
    - main

Docker Build Pipeline

stages:
  - build
  - test
  - release

variables:
  DOCKER_DRIVER: overlay2
  IMAGE_TAG: $CI_REGISTRY_IMAGE:$CI_COMMIT_REF_SLUG

build:
  stage: build
  image: docker:latest
  services:
    - docker:dind
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build -t $IMAGE_TAG .
    - docker push $IMAGE_TAG

test:
  stage: test
  image: $IMAGE_TAG
  script:
    - npm test

release:
  stage: release
  image: docker:latest
  services:
    - docker:dind
  script:
    - docker pull $IMAGE_TAG
    - docker tag $IMAGE_TAG $CI_REGISTRY_IMAGE:latest
    - docker push $CI_REGISTRY_IMAGE:latest
  only:
    - main

Multi-Stage Pipeline with Environments

stages:
  - build
  - test
  - staging
  - production

build:
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm test

deploy:staging:
  stage: staging
  script:
    - ./deploy.sh staging
  environment:
    name: staging
    url: https://staging.myapp.com
  only:
    - develop

deploy:production:
  stage: production
  script:
    - ./deploy.sh production
  environment:
    name: production
    url: https://myapp.com
  when: manual  # Requires manual approval
  only:
    - main

Auto DevOps

# Enable Auto DevOps in Settings > CI/CD > Auto DevOps
# GitLab automatically:
# - Builds Docker images
# - Runs tests
# - Performs security scanning
# - Deploys to Kubernetes
# - Sets up monitoring

# No .gitlab-ci.yml needed for standard apps!

# Customize by adding .gitlab-ci.yml with only specific jobs
include:
  - template: Auto-DevOps.gitlab-ci.yml

# Override specific jobs
test:
  script:
    - npm test -- --custom-flag

🚀 Advanced Topics

GitLab Runner Configuration

# config.toml for GitLab Runner
concurrent = 4

[[runners]]
  name = "docker-runner"
  url = "https://gitlab.com/"
  token = "your-runner-token"
  executor = "docker"
  [runners.docker]
    image = "node:20"
    privileged = false
    volumes = ["/cache"]
  [runners.cache]
    Type = "s3"
    Shared = true

Kubernetes Deployment

# Deploy to Kubernetes cluster
deploy:
  stage: deploy
  image: bitnami/kubectl:latest
  script:
    - kubectl config set-cluster k8s --server="$KUBE_URL" --insecure-skip-tls-verify=true
    - kubectl config set-credentials admin --token="$KUBE_TOKEN"
    - kubectl config set-context default --cluster=k8s --user=admin
    - kubectl config use-context default
    - kubectl apply -f k8s/deployment.yaml
    - kubectl rollout status deployment/myapp
  environment:
    name: production
    kubernetes:
      namespace: production

Scheduled Pipelines (Cron Jobs)

# Create in CI/CD > Schedules
# Or define in pipeline:

nightly-build:
  script:
    - npm run build
  only:
    - schedules

# Use variables to distinguish scheduled runs
cleanup:
  script:
    - ./cleanup.sh
  only:
    variables:
      - $CI_PIPELINE_SOURCE == "schedule"
