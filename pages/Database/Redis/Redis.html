📚 Introduction

What is Redis?

Redis (Remote Dictionary Server) is an open-source, in-memory data structure store that can be used as a database, cache, message broker, and streaming engine. Known for its exceptional performance, Redis delivers sub-millisecond response times enabling millions of requests per second for real-time applications in gaming, fintech, ad-tech, social media, and IoT.

Key Features

⚡ Blazing Fast — Sub-millisecond latency, millions of ops/sec
🔧 Rich Data Structures — Strings, Lists, Sets, Hashes, Sorted Sets, Streams
💾 Persistence Options — RDB snapshots and AOF logging
🔄 Replication — Master-replica replication for high availability
📡 Pub/Sub — Message broadcasting and channels
🌊 Streams — Append-only log for event sourcing
⏰ Expiration & TTL — Automatic key expiration
🔒 Transactions — MULTI/EXEC for atomic operations

Why Choose Redis?

  • Performance: All data resides in memory, providing ultra-low latency for both reads and writes.
  • Versatility: Beyond simple key-value storage, Redis supports complex data structures enabling sophisticated use cases.
  • Simplicity: Easy to deploy, manage, and scale with a straightforward command set.
  • Reliability: Persistence options, replication, and Redis Sentinel for automatic failover ensure data durability.
  • Scalability: Redis Cluster provides automatic partitioning across multiple nodes.
  • Ecosystem: Extensive client libraries for all major programming languages and active community support.

Use Cases

🔥 Caching — Database query results, API responses, session data
🔐 Session Store — User sessions for web applications
📊 Real-Time Analytics — Leaderboards, counters, metrics
⏱️ Rate Limiting — API rate limiting with expiration
📬 Message Queue — Task queues, job processing
🎮 Gaming — Leaderboards, player state, matchmaking
🔔 Notifications — Real-time pub/sub messaging
🌐 Geospatial — Location-based services

Redis Data Structures

  • Strings: Binary-safe strings, integers, bitmaps (max 512MB)
  • Lists: Linked lists for queues and stacks (LPUSH, RPOP, etc.)
  • Sets: Unordered collections of unique strings with set operations
  • Hashes: Field-value pairs, perfect for objects
  • Sorted Sets: Sets with scores for rankings and leaderboards
  • Streams: Append-only logs for event sourcing and message queues
  • HyperLogLog: Probabilistic data structure for cardinality estimation
  • Geospatial: Store and query coordinates
  • Bitmaps & Bitfields: Efficient bit-level operations

Persistence Models

  • RDB (Redis Database): Point-in-time snapshots at specified intervals
  • AOF (Append Only File): Log of every write operation for durability
  • Hybrid: Combine both for best of both worlds
  • No Persistence: Pure cache mode for maximum performance
⚠️ Important: Redis is single-threaded (for command execution) which ensures simplicity and atomic operations. Use Redis for what it does best - fast in-memory operations - and combine with persistent databases for long-term storage.

⚙️ Installation & Setup

Get started with Redis on your platform. Redis is available for Linux, macOS, and can run on Windows via WSL2 or Docker.

Windows Installation

📌 Note: Redis does not officially support Windows. Use WSL2 (recommended) or Docker for development.

Option 1 — WSL2 (Recommended)

# Install WSL2 with Ubuntu
wsl --install -d Ubuntu

# Start WSL2
wsl

# Inside WSL2, update packages
sudo apt update

# Install Redis
sudo apt install redis-server

# Configure Redis to use systemd
sudo sed -i 's/^supervised no/supervised systemd/' /etc/redis/redis.conf

# Start Redis service
sudo systemctl enable redis-server
sudo systemctl start redis-server

# Check status
sudo systemctl status redis-server

# Test connection
redis-cli ping
# Should return: PONG

Option 2 — Docker (Alternative)

# Pull and run Redis
docker run -d --name redis -p 6379:6379 redis:7-alpine

# Connect to Redis CLI
docker exec -it redis redis-cli

# View logs
docker logs redis

macOS Installation

# Install Redis using Homebrew
brew install redis

# Start Redis as a service
brew services start redis

# Verify service is running
brew services list

# Stop service
brew services stop redis

# Run Redis in foreground (for testing)
redis-server /usr/local/etc/redis.conf

# Connect to Redis CLI
redis-cli

# Test connection

# Should return: PONG

  --name redis \

# Run with password protection
docker run -d \
  --name redis \
  -p 6379:6379 \
  -v redis_data:/data \
  redis:7-alpine redis-server --requirepass StrongP@ssw0rd

# Run with custom config
docker run -d \
  --name redis \
  -p 6379:6379 \
  -v $(pwd)/redis.conf:/usr/local/etc/redis/redis.conf \
  -v redis_data:/data \
  redis:7-alpine redis-server /usr/local/etc/redis/redis.conf

# Access Redis CLI
docker exec -it redis redis-cli

# With password
docker exec -it redis redis-cli -a StrongP@ssw0rd

# Stop and remove
docker stop redis && docker rm redis

Configuration & Security

Basic Configuration (redis.conf)

# Bind to specific interface
bind 127.0.0.1 ::1      # Localhost only (default, secure)
# bind 0.0.0.0          # All interfaces (development only)

# Port
port 6379

# Password protection (highly recommended)
requirepass StrongP@ssw0rd

# Maximum memory
maxmemory 256mb
maxmemory-policy allkeys-lru  # Eviction policy

# Persistence - RDB
save 900 1              # Save after 900 sec if 1 key changed
save 300 10             # Save after 300 sec if 10 keys changed
save 60 10000           # Save after 60 sec if 10000 keys changed

# Persistence - AOF
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec    # fsync every second (balanced)

# Logging
loglevel notice
logfile /var/log/redis/redis-server.log

# After editing, restart Redis
sudo systemctl restart redis-server

Connect with Password

# Connect and authenticate
redis-cli
127.0.0.1:6379> AUTH StrongP@ssw0rd

# Or provide password in connection
redis-cli -a StrongP@ssw0rd

# Connection string for applications
redis://default:StrongP@ssw0rd@localhost:6379

📖 User Guide

Learn essential Redis commands and patterns for building high-performance applications.

Strings - Basic Key-Value Operations

# Set a key
SET user:1:name "Ada Lovelace"
SET user:1:email "ada@example.com"

# Get a key
GET user:1:name
# Returns: "Ada Lovelace"

# Set with expiration (in seconds)
SETEX session:abc123 3600 "user_data"  # Expires in 1 hour
SET cache:key "value" EX 300           # Alternative syntax

# Set if not exists
SETNX user:1:name "Grace Hopper"
# Returns: 0 (key already exists)

# Set multiple keys at once
MSET user:2:name "Alan Turing" user:2:email "alan@example.com"

# Get multiple keys
MGET user:1:name user:2:name

# Increment/Decrement (atomic)
SET visits:homepage 0
INCR visits:homepage       # Returns: 1
INCR visits:homepage       # Returns: 2
INCRBY visits:homepage 10  # Returns: 12
DECR visits:homepage       # Returns: 11

# Append to string
SET greeting "Hello"
APPEND greeting " World"
GET greeting
# Returns: "Hello World"

# Get string length
STRLEN greeting
# Returns: 11

# Delete key
DEL user:1:name

# Check if key exists
EXISTS user:1:email
# Returns: 1 (exists) or 0 (doesn't exist)

# Set expiration on existing key
EXPIRE user:1:email 3600

# Check time to live
TTL user:1:email
# Returns: seconds remaining or -1 (no expiration) or -2 (doesn't exist)

# Remove expiration
PERSIST user:1:email

Lists - Ordered Collections

# Push to list (left/right)
LPUSH tasks "task3"    # Add to beginning
LPUSH tasks "task2"
LPUSH tasks "task1"
RPUSH tasks "task4"    # Add to end

# View list (0 to -1 means all elements)
LRANGE tasks 0 -1
# Returns: ["task1", "task2", "task3", "task4"]

# Get list length
LLEN tasks

# Pop from list
LPOP tasks             # Remove and return first element
RPOP tasks             # Remove and return last element

# Blocking pop (wait for element)
BLPOP tasks 30         # Wait 30 seconds for element
BRPOP tasks 0          # Wait indefinitely

# Get element by index
LINDEX tasks 0         # First element
LINDEX tasks -1        # Last element

# Set element at index
LSET tasks 0 "updated_task"

# Remove elements
LREM tasks 1 "task2"   # Remove first occurrence

# Trim list to specified range
LTRIM tasks 0 99       # Keep first 100 elements

# Use case: Task Queue
LPUSH queue:jobs '{"task": "send_email", "user_id": 123}'
RPOP queue:jobs        # Process oldest job

Sets - Unique Unordered Collections

# Add members to set
SADD tags:article1 "redis" "database" "nosql"
SADD tags:article2 "redis" "cache" "performance"

# Get all members
SMEMBERS tags:article1

# Check if member exists
SISMEMBER tags:article1 "redis"
# Returns: 1 (yes) or 0 (no)

# Remove member
SREM tags:article1 "nosql"

# Get set size
SCARD tags:article1

# Set operations
# Union (tags in either article)
SUNION tags:article1 tags:article2

# Intersection (common tags)
SINTER tags:article1 tags:article2

# Difference (tags in article1 but not article2)
SDIFF tags:article1 tags:article2

# Random member
SRANDMEMBER tags:article1      # Get random member
SPOP tags:article1             # Remove and return random member

# Use case: Unique visitors
SADD visitors:2025-12-23 "user:123" "user:456"
SCARD visitors:2025-12-23      # Count unique visitors

Hashes - Field-Value Pairs (Objects)

# Set hash fields
HSET user:1 name "Ada Lovelace"
HSET user:1 email "ada@example.com"
HSET user:1 age 30

# Set multiple fields at once
HMSET user:2 name "Grace Hopper" email "grace@example.com" age 85

# Get field value
HGET user:1 name

# Get multiple fields
HMGET user:1 name email

# Get all fields and values
HGETALL user:1
# Returns: ["name", "Ada Lovelace", "email", "ada@example.com", "age", "30"]

# Get all field names
HKEYS user:1

# Get all values
HVALS user:1

# Check if field exists
HEXISTS user:1 name

# Delete field
HDEL user:1 age

# Get number of fields
HLEN user:1

# Increment field value
HINCRBY user:1 login_count 1

# Use case: Session storage
HMSET session:abc123 user_id 1 username "ada" last_seen 1703350000
EXPIRE session:abc123 3600
HGETALL session:abc123

Sorted Sets - Ordered by Score

# Add members with scores
ZADD leaderboard 100 "player1"
ZADD leaderboard 85 "player2"
ZADD leaderboard 92 "player3"

# Add multiple at once
ZADD leaderboard 78 "player4" 95 "player5"

# Get members by rank (lowest to highest score)
ZRANGE leaderboard 0 -1

# Get members with scores
ZRANGE leaderboard 0 -1 WITHSCORES

# Get top 3 (highest scores)
ZREVRANGE leaderboard 0 2 WITHSCORES

# Get score of member
ZSCORE leaderboard "player1"

# Get rank of member (0-based)
ZRANK leaderboard "player1"        # Ascending
ZREVRANK leaderboard "player1"     # Descending (for leaderboards)

# Increment score
ZINCRBY leaderboard 5 "player1"    # Add 5 points

# Count members in score range
ZCOUNT leaderboard 80 100

# Get members by score range
ZRANGEBYSCORE leaderboard 80 100

# Remove member
ZREM leaderboard "player4"

# Remove by rank range
ZREMRANGEBYRANK leaderboard 0 0    # Remove lowest

# Remove by score range
ZREMRANGEBYSCORE leaderboard 0 70  # Remove all below 70

# Get sorted set size
ZCARD leaderboard

# Use case: Real-time leaderboard
ZADD game:leaderboard 15420 "user:123" 12890 "user:456"
ZREVRANGE game:leaderboard 0 9 WITHSCORES  # Top 10 players

Common Patterns & Use Cases

1. Caching Pattern

# Cache database query result
SET cache:user:123 '{"id":123,"name":"Ada"}' EX 300

# In application:
# 1. Check Redis cache
# 2. If miss, query database
# 3. Store result in Redis with expiration
GET cache:user:123
# If null, fetch from DB and:
SET cache:user:123 '{"data":"..."}' EX 300

2. Rate Limiting

# Simple rate limiter (10 requests per minute)
SET rate:user:123:2025-12-23-22 0 EX 60 NX
INCR rate:user:123:2025-12-23-22
GET rate:user:123:2025-12-23-22

# In application:
# count = INCR rate:user:{id}:{current_minute}
# if count > 10: reject request

3. Session Storage

# Store session data
HMSET session:abc123 \
  user_id 1 \
  username "ada" \
  role "admin" \
  created_at 1703350000

# Set expiration (30 minutes)
EXPIRE session:abc123 1800

# Extend session on activity
EXPIRE session:abc123 1800

4. Counters & Analytics

# Page views counter
INCR pageviews:homepage
INCR pageviews:about

# Unique visitors (using sets)
SADD visitors:2025-12-23 "user:123"
SADD visitors:2025-12-23 "user:456"
SCARD visitors:2025-12-23  # Count unique visitors

🚀 Advanced Features

Pub/Sub - Messaging

# Subscribe to channel (in one client)
SUBSCRIBE notifications

# Publish message (in another client)
PUBLISH notifications "New message from Ada"

# Pattern subscribe (wildcard)
PSUBSCRIBE user:*:notifications

# Publish to pattern
PUBLISH user:123:notifications "You have a new follower"

# Unsubscribe
UNSUBSCRIBE notifications

Streams - Event Sourcing

# Add entry to stream
XADD events:clickstream * user_id 123 action "view" page "homepage"

# Add with explicit ID
XADD events:clickstream 1703350000000-0 event "purchase" amount 99.99

# Read stream entries
XRANGE events:clickstream - +       # All entries
XRANGE events:clickstream - + COUNT 10  # First 10

# Read new entries (blocking)
XREAD BLOCK 0 STREAMS events:clickstream $

# Consumer groups
XGROUP CREATE events:clickstream analytics $ MKSTREAM
XREADGROUP GROUP analytics consumer1 STREAMS events:clickstream >

# Get stream length
XLEN events:clickstream

Transactions

# Basic transaction
MULTI
SET key1 "value1"
SET key2 "value2"
INCR counter
EXEC

# Transaction with watch (optimistic locking)
WATCH balance:user1
GET balance:user1
# Check if balance is sufficient
MULTI
DECRBY balance:user1 100
INCRBY balance:user2 100
EXEC

# Discard transaction
MULTI
SET key "value"
DISCARD  # Cancel transaction

Lua Scripting

# Execute Lua script
EVAL "return redis.call('SET', KEYS[1], ARGV[1])" 1 mykey "myvalue"

# Atomic rate limiter script
EVAL "
  local key = KEYS[1]
  local limit = tonumber(ARGV[1])
  local current = tonumber(redis.call('GET', key) or '0')
  if current + 1 > limit then
    return 0
  else
    redis.call('INCR', key)
    redis.call('EXPIRE', key, 60)
    return 1
  end
" 1 rate:user:123 10

# Load script and use SHA
SCRIPT LOAD "return redis.call('GET', KEYS[1])"
# Returns SHA: abc123...
EVALSHA abc123... 1 mykey
