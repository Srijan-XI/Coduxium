📚 Introduction

What is Amazon DynamoDB?

Amazon DynamoDB is AWS's flagship fully managed NoSQL database service that provides fast and predictable performance with seamless scalability. It's designed to handle massive workloads with automatic scaling, built-in security, backup and restore capabilities, and in-memory caching.

Key Features

⚡ Performance — Single-digit millisecond latency at any scale
🚀 Serverless — No servers to manage, fully managed service
🌍 Global Tables — Multi-region, active-active replication
📈 Auto Scaling — Automatic capacity adjustment based on traffic
💰 Flexible Pricing — Provisioned or on-demand capacity modes
🔒 Security — Integrated with AWS IAM, encryption at rest and in transit
⚙️ DAX — In-memory acceleration for microsecond latency
📊 Streams — Change data capture for event-driven architecture

Why Choose DynamoDB?

  • Performance: Delivers consistent single-digit millisecond response times regardless of scale, making it ideal for high-traffic applications.
  • Operationally Simple: No database administration required. AWS handles hardware provisioning, setup, configuration, replication, software patching, and cluster scaling.
  • Cost Efficient: Pay only for what you use with on-demand pricing, or reserve capacity for predictable workloads at lower cost.
  • Global Reach: Global Tables provide multi-region, fully replicated, high-performance databases for globally distributed applications.
  • Event-Driven: DynamoDB Streams enables real-time processing of data changes for analytics, notifications, or materialized views.
  • ACID Transactions: Support for transactions across multiple items and tables for complex business logic.

Use Cases

🎮 Gaming — Player data, leaderboards, session state
🛒 E-commerce — Shopping carts, product catalogs, order history
📱 Mobile/Web Apps — User profiles, preferences, activity feeds
🌐 IoT — Device data, telemetry, time-series data
📰 Content — Metadata, user-generated content, comments
💳 Financial — Transaction logs, fraud detection, trading data
⚠️ Important: DynamoDB is not a relational database. Design your data model around your access patterns, not around normalized tables with joins.

📖 User Guide

Learn the core concepts and best practices for working with DynamoDB effectively.

Core Concepts

Table — Collection of items (like a table in SQL)
Item — Collection of attributes (like a row)
Attribute — Key-value pair (like a column)
Partition Key (PK) — Hash key for data distribution
Sort Key (SK) — Range key for sorting items
GSI — Global Secondary Index (alternate query pattern)
LSI — Local Secondary Index (alternate sort key)
TTL — Time-To-Live for automatic item deletion
Streams — Change data capture for events
Transactions — ACID operations across items

Data Modeling Best Practices

1. Single-Table Design

Store multiple entity types in one table using composite keys and type prefixes. This allows you to fetch related data in a single query.

{
  "PK": "USER#42",
  "SK": "PROFILE#2025-01-01",
  "Type": "UserProfile",
  "Name": "Ada Lovelace",
  "Email": "ada@example.com",
  "City": "London"
}
{
  "PK": "USER#42",
  "SK": "ORDER#A1001",
  "Type": "Order",
  "OrderDate": "2025-12-20",
  "Total": 79.90,
  "Status": "shipped"
}
{
  "PK": "USER#42",
  "SK": "ORDER#A1002",
  "Type": "Order",
  "OrderDate": "2025-12-22",
  "Total": 125.50,
  "Status": "processing"
}

2. Access Patterns First

List all your access patterns before designing your schema:

  • ✅ Get user profile by user ID
  • ✅ Get all orders for a user (sorted by date)
  • ✅ Get order details by order ID
  • ✅ Get all users in a city (requires GSI)
  • ✅ Get recent orders across all users (requires GSI)

3. Partition Key Selection

  • High Cardinality: Choose a partition key with many unique values (user ID, product ID, order ID)
  • Even Distribution: Ensure data is evenly distributed across partitions
  • Avoid Hot Partitions: Don't use low-cardinality keys like status, country, or date

4. Sort Key Patterns

// Hierarchical data
PK: "USER#42"
SK: "PROFILE#METADATA"
SK: "ORDER#2025-12-20#A1001"
SK: "ADDRESS#HOME"
SK: "ADDRESS#WORK"

// Time-series data
PK: "SENSOR#S123"
SK: "2025-12-23T10:30:00Z"
SK: "2025-12-23T10:31:00Z"

// Relationships
PK: "PRODUCT#P456"
SK: "REVIEW#USER#42#2025-12-20"
SK: "REVIEW#USER#99#2025-12-21"
⚠️ Design Rule: Design for your access patterns, not for how the data looks. NoSQL requires a different mindset than relational databases.

CRUD Operations

Python (Boto3) Examples

import boto3
from boto3.dynamodb.conditions import Key, Attr

# Initialize DynamoDB client
dynamodb = boto3.resource('dynamodb', region_name='us-east-1')
table = dynamodb.Table('users')

# CREATE - Put Item
table.put_item(
    Item={
        'PK': 'USER#42',
        'SK': 'PROFILE#2025',
        'Name': 'Ada Lovelace',
        'Email': 'ada@example.com',
        'City': 'London',
        'CreatedAt': '2025-12-23T10:30:00Z'
    }
)

# READ - Get Item
response = table.get_item(
    Key={
        'PK': 'USER#42',
        'SK': 'PROFILE#2025'
    }
)
item = response.get('Item')
print(item)

# UPDATE - Update Item
table.update_item(
    Key={
        'PK': 'USER#42',
        'SK': 'PROFILE#2025'
    },
    UpdateExpression='SET City = :city, Email = :email',
    ExpressionAttributeValues={
        ':city': 'Paris',
        ':email': 'ada.lovelace@example.com'
    }
)

# DELETE - Delete Item
table.delete_item(
    Key={
        'PK': 'USER#42',
        'SK': 'PROFILE#2025'
    }
)

# QUERY - Get all items with same partition key
response = table.query(
    KeyConditionExpression=Key('PK').eq('USER#42')
)
items = response['Items']

# QUERY with Sort Key condition
response = table.query(
    KeyConditionExpression=Key('PK').eq('USER#42') & Key('SK').begins_with('ORDER#')
)
orders = response['Items']

# SCAN with filter (expensive - avoid in production)
response = table.scan(
    FilterExpression=Attr('City').eq('London')
)
london_users = response['Items']

Node.js (AWS SDK v3) Examples

import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import { 
  DynamoDBDocumentClient, 
  PutCommand,

  UpdateCommand,
  DeleteCommand,

  ScanCommand

// Initialize client
const client = new DynamoDBClient({ region: "us-east-1" });
const docClient = DynamoDBDocumentClient.from(client);

// CREATE - Put Item
await docClient.send(new PutCommand({

    City: "London",
    CreatedAt: new Date().toISOString()
  }
}));

// READ - Get Item
const { Item } = await docClient.send(new GetCommand({
  TableName: "users",
  Key: {
    PK: "USER#42",
    SK: "PROFILE#2025"
  }
}));
console.log(Item);

// UPDATE - Update Item
await docClient.send(new UpdateCommand({
  TableName: "users",
  Key: {
    PK: "USER#42",
    SK: "PROFILE#2025"
  },
  UpdateExpression: "SET City = :city, Email = :email",
  ExpressionAttributeValues: {
    ":city": "Paris",
    ":email": "ada.lovelace@example.com"

Secondary Indexes

Global Secondary Index (GSI)

Allows you to query on non-key attributes. GSI has its own partition and sort keys.

# Create table with GSI
aws dynamodb create-table \
  --table-name users \
  --attribute-definitions \
    AttributeName=PK,AttributeType=S \
    AttributeName=SK,AttributeType=S \
    AttributeName=Email,AttributeType=S \
    AttributeName=City,AttributeType=S \
  --key-schema \
    AttributeName=PK,KeyType=HASH \
    AttributeName=SK,KeyType=RANGE \
  --global-secondary-indexes \
    '[{
      "IndexName": "EmailIndex",
      "KeySchema": [{"AttributeName":"Email","KeyType":"HASH"}],
      "Projection": {"ProjectionType":"ALL"},
      "ProvisionedThroughput": {"ReadCapacityUnits":5,"WriteCapacityUnits":5}
    },
    {
      "IndexName": "CityIndex",
      "KeySchema": [
        {"AttributeName":"City","KeyType":"HASH"},
        {"AttributeName":"SK","KeyType":"RANGE"}
      ],
      "Projection": {"ProjectionType":"ALL"},
      "ProvisionedThroughput": {"ReadCapacityUnits":5,"WriteCapacityUnits":5}
    }]' \
  --billing-mode PROVISIONED \
  --provisioned-throughput ReadCapacityUnits=5,WriteCapacityUnits=5

# Query using GSI
aws dynamodb query \
  --table-name users \
  --index-name EmailIndex \
  --key-condition-expression "Email = :email" \
  --expression-attribute-values '{":email": {"S": "ada@example.com"}}'

Local Secondary Index (LSI)

Alternative sort key for the same partition key. Must be created at table creation time.

# Create table with LSI
aws dynamodb create-table \
  --table-name users \
  --attribute-definitions \
    AttributeName=PK,AttributeType=S \
    AttributeName=SK,AttributeType=S \
    AttributeName=CreatedAt,AttributeType=S \
  --key-schema \
    AttributeName=PK,KeyType=HASH \
    AttributeName=SK,KeyType=RANGE \
  --local-secondary-indexes \
    '[{
      "IndexName": "CreatedAtIndex",
      "KeySchema": [
        {"AttributeName":"PK","KeyType":"HASH"},
        {"AttributeName":"CreatedAt","KeyType":"RANGE"}
      ],
      "Projection": {"ProjectionType":"ALL"}
    }]' \
  --billing-mode PAY_PER_REQUEST

Advanced Features

Transactions

Perform ACID operations across multiple items:

# Python - Transaction example
import boto3

client = boto3.client('dynamodb')

response = client.transact_write_items(
    TransactItems=[
        {
            'Put': {
                'TableName': 'users',
                'Item': {
                    'PK': {'S': 'USER#100'},
                    'SK': {'S': 'PROFILE'},
                    'Balance': {'N': '1000'}
                }
            }
        },
        {
            'Update': {
                'TableName': 'users',
                'Key': {
                    'PK': {'S': 'USER#99'},
                    'SK': {'S': 'PROFILE'}
                },
                'UpdateExpression': 'SET Balance = Balance - :amount',
                'ExpressionAttributeValues': {
                    ':amount': {'N': '100'}
                }
            }
        }
    ]
)

Time To Live (TTL)

Automatically delete items after a specific time:

# Enable TTL on a table
aws dynamodb update-time-to-live \
  --table-name users \
  --time-to-live-specification "Enabled=true,AttributeName=ExpiresAt"

# Add item with TTL (expires in 30 days)
import time
expiration_time = int(time.time()) + (30 * 24 * 60 * 60)

table.put_item(
    Item={
        'PK': 'SESSION#abc123',
        'SK': 'DATA',
        'UserId': 'USER#42',
        'ExpiresAt': expiration_time
    }
)

DynamoDB Streams

Capture changes to table items:

# Enable streams on a table
aws dynamodb update-table \
  --table-name users \
  --stream-specification StreamEnabled=true,StreamViewType=NEW_AND_OLD_IMAGES

# Process streams with Lambda (Python)
def lambda_handler(event, context):
    for record in event['Records']:
        if record['eventName'] == 'INSERT':
            new_image = record['dynamodb']['NewImage']
            print(f"New item: {new_image}")
        elif record['eventName'] == 'MODIFY':
            old_image = record['dynamodb']['OldImage']
            new_image = record['dynamodb']['NewImage']
            print(f"Updated: {old_image} -> {new_image}")
        elif record['eventName'] == 'REMOVE':
            old_image = record['dynamodb']['OldImage']
            print(f"Deleted: {old_image}")

Best Practices

  • ✅ Model for access patterns: Define how your application will query data before designing your schema
  • ✅ Use single-table design: Store related entities in one table to minimize round trips
  • ✅ Avoid scans: Use Query operations with partition keys instead of Scan whenever possible
  • ✅ Use sparse indexes: GSIs only index items that have the index key attributes
  • ✅ Enable point-in-time recovery: Protects against accidental deletes and data corruption
  • ✅ Use on-demand mode initially: Switch to provisioned when traffic becomes predictable
  • ✅ Implement retry logic: Handle throttling errors with exponential backoff
  • ✅ Monitor with CloudWatch: Track ConsumedReadCapacity, ConsumedWriteCapacity, and ThrottledRequests
  • ❌ Don't use DynamoDB as a relational database: No joins, foreign keys, or complex queries
  • ❌ Don't store large items: Max item size is 400KB; use S3 for large objects
